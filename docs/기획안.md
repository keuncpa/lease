# LeaseLens 기획안
### 감사인 관점의 IFRS 16(K-IFRS 1116) 리스 계약서 AI 추출·독립재계산·검증 플랫폼

> 작성 목적: 회계법인 디지털 전형 지원용 포트폴리오. 기존 conversion / XBRL 프로젝트에 이은 세 번째 작품으로, GitHub + Vercel 배포를 전제로 한다.
> 핵심 메시지: **"리스 계산을 자동화하는 도구"는 이미 시장에 많다. 나는 그 위에 *감사인이 신뢰할 수 있는 근거·검증 레이어*를 얹어 차별화한다.**

---

## 1. 배경과 문제 정의

K-IFRS 제1116호(리스, 2019년 시행)는 리스이용자에게 **운용/금융 구분을 없앤 단일모형**을 요구한다. 거의 모든 리스를 사용권자산·리스부채로 재무상태표에 인식해야 하므로, 기업과 감사인 모두 리스별 현재가치·상각표·재측정을 반복 산출해야 한다.

문제는 이 과정이 여전히 **엑셀과 수작업에 의존**한다는 점이다. 리스 포트폴리오가 커지면 스프레드시트는 수식 오류·버전 충돌·감사증적 부재에 취약해지고, 할인율·리스기간·지급조건의 작은 입력 오류가 큰 왜곡표시로 이어진다. (출처: Trullion, MRI Software 등 — `참고자료` 절)

대형 회계법인과 리스 솔루션 업체들은 디지털 솔루션(로보틱 자동화, 데이터 뷰어 계열 등)과 리스 회계 자문을 제공하지만, 공개된 리스 솔루션 자료는 대체로 **"계산·전기 자동화"**에 초점이 맞춰져 있다. 본 프로젝트는 그 자동화의 **하류(下流)에서 감사인이 실제로 부딪히는 검증 문제**를 정면으로 다룬다.

## 2. 기존 리스 자동화 솔루션의 한계 (보완 대상)

| # | 기존 솔루션의 일반적 한계 | 감사 현장에서의 의미 |
|---|---|---|
| L1 | **입력의 출처가 끊긴다.** 계약서에서 숫자를 사람이 옮겨 적고, 시스템은 그 숫자만 받는다. | 감사인이 "이 리스료·할인율이 계약서 어디서 나왔나"를 다시 원본으로 역추적해야 함 (근거 추적성 부재) |
| L2 | **계산은 하지만 검증은 안 한다.** 입력이 맞다고 가정하고 PV·상각표를 산출. | 할인율 이상치, 누락 항목, 회사 수치와의 차이를 시스템이 잡아주지 않음 |
| L3 | **판단 영역을 자동화에서 배제.** 연장/매수선택권, 변동리스료, 복구의무 등은 "사람이 알아서". | 리스기간·측정범위 판단 누락이 가장 흔한 왜곡 원인인데 플래그가 없음 |
| L4 | **가정이 블랙박스.** 할인율 기간환산(명목 vs 유효), 선급/후급 처리 등이 내부에 숨음. | 재계산 차이가 났을 때 원인 분해가 어려움 |
| L5 | **감사조서로 바로 안 이어진다.** 결과는 숫자일 뿐, 조서·발견사항·권고절차로 구조화되지 않음. | 자동화 산출물과 감사문서 사이에 수작업 갭 |

> 핵심 통찰: 시장의 리스 솔루션은 **"작성자(preparer) 도구"**다. 나는 같은 데이터 위에서 동작하는 **"감사인(auditor) 도구"**를 만든다. 이것이 기존 솔루션과 동일하게 만들지 않으면서 고도화하는 지점이다.

## 3. 차별화 컨셉 — 5개의 보완 축

1. **AI 계약서 추출 + 근거 추적성 (L1 보완)**
   - 계약서 텍스트/PDF에서 리스 조건(당사자·개시일·기간·리스료·주기·할인율·선택권·변동리스료·복구의무)을 자동 추출.
   - 모든 추출 필드가 `{값 + 신뢰도(0~1) + 근거조항 원문}`을 함께 보유. → 감사인이 클릭 한 번으로 "이 숫자의 출처"를 본다.

2. **독립 재계산 엔진 (L2·L4 보완)**
   - 추출값으로 리스부채(PV)·사용권자산·상각표·이자/감가상각을 **회사와 독립적으로** 재계산.
   - 회사 제시 수치와 비교해 허용오차(기본 1%) 초과 시 자동 플래그.
   - 할인율 기간환산 방식(유효/명목), 선급/후급을 **명시적으로 노출** → 가정 블랙박스 제거.

3. **이상탐지 / 위험 플래그 (L2·L3 보완)**
   - 완전성(필수항목 누락·저신뢰), 평가(할인율 합리성 1~15% 밴드, 선택권 영향), 표시·공시(변동리스료 분리), 가정(지급시기 불명확), 복구원가 누락 등을 규칙으로 점검.
   - 각 발견사항에 **심각도 + 분류 + 권고 감사절차**를 부여.

4. **판단 영역의 가시화 (L3 보완)**
   - 연장/매수선택권 → "리스기간에 반영했는가" 질문 자동 생성.
   - 변동리스료 → "지수연동분만 측정 포함, 실적연동분 비용처리" 가이드.
   - 복구의무 → "사용권자산 가산 누락" 점검.

5. **감사조서 자동 생성 (L5 보완)**
   - 추출표(근거 포함) + 재계산 + 발견사항 + 상각표를 하나의 **추적 가능한 조서(.md)**로 내보내기.
   - "자동화 산출물 → 감사문서" 갭 제거.

## 4. 시스템 아키텍처

```
계약서(텍스트/PDF)
      │
      ▼
[0] PDF 레이어  api/parse-pdf · extraction/pdf.ts
      ├─ 디지털 PDF → 서버 텍스트 레이어 추출(pdfjs-dist)
      └─ 스캔본 감지 → 브라우저 OCR(tesseract.js 한/영) 폴백
      │   → 추출된 텍스트를 [1]로 전달
      ▼
[1] 추출 레이어  extraction/
      ├─ 규칙기반(extractRuleBased)  ← API키 불필요, 데모/오프라인
      └─ LLM(api/extract route)      ← ANTHROPIC_API_KEY, 동일 스키마 JSON
      │   결과: ExtractedLease { 값 + 신뢰도 + 근거조항 }
      ▼
[2] 계산 엔진   ifrs16/engine.ts   ← 단위테스트 6종 통과
      ├─ presentValueOfPayments / buildSchedule
      └─ 리스부채·사용권자산·상각표·분개
      ▼
[3] 감사 검증   audit/checks.ts
      ├─ recalc()    독립 재계산 vs 회사수치
      └─ runChecks() 이상탐지·위험 플래그
      ▼
[4] 산출        audit/workpaper.ts + app/page.tsx
      ├─ 대시보드(추출표·KPI·상각표·분개·발견사항)
      └─ 감사조서 .md 내보내기
```

설계 원칙: 계산 엔진은 **외부 의존성 0, 결정론적, 모든 중간값 노출**(감사 재계산 가능). 추출 레이어는 규칙기반/LLM 어느 쪽이든 **동일한 스키마**를 반환하므로 다운스트림이 동일하게 동작.

## 5. 화면 구성 (구현됨)

- **좌측 입력 패널**: 시작은 **빈 상태**. **PDF 드래그&드롭/업로드**(텍스트 레이어 추출 + 스캔본 OCR 폴백) 또는 **예시 계약서 4종**(정상 / 위험 다수 / 복구의무 / 회사수치 불일치) 불러오기로 계약서를 명시적으로 입력해야 한다. 입력된 계약서는 출처 배지(PDF/예시/직접입력)와 함께 편집 가능한 텍스트로 표시되며, **입력이 없으면 '추출 → 분석' 버튼이 비활성화**되어 빈 상태에서는 분석이 실행되지 않는다. LLM 사용 토글 제공.
- **우측 결과 패널**:
  - KPI 카드 — 개시 리스부채 / 사용권자산 / 위험 플래그(High·Med·Low)
  - 추출 데이터 테이블 — 항목·값·**신뢰도 배지**·근거조항
  - 독립 재계산 검증 — 기간이자율·엔진값·회사값·차이(PASS/FAIL)
  - 위험 플래그 카드 — 심각도·분류·내용·권고절차
  - 상각표 미리보기 + 개시 분개 + **감사조서 내보내기 버튼**

## 6. 기술 스택 & 배포

- **Next.js 14 (App Router) + TypeScript + Tailwind CSS** — Vercel 1-클릭 배포 정합.
- 계산 엔진은 순수 TS, **Node 내장 테스트 러너**로 검증(`npm test`).
- LLM 추출은 **Edge Function**(`/api/extract`), `ANTHROPIC_API_KEY` 환경변수. 키가 없으면 자동으로 규칙기반 폴백 → **배포 즉시 키 없이도 데모 가능**.
- GitHub → Vercel 연결 시 `next build` 그대로 통과(빌드 검증 완료).

## 7. 회계감사 진로와의 연결성 (면접 어필 포인트)

> 나의 목표는 회계감사다. 이 프로젝트는 "개발을 할 줄 안다"가 아니라 **"감사인의 문제를 코드로 정의할 줄 안다"**를 보여준다.

- **감사인의 사고를 제품화했다.** 완전성·정확성·평가·표시공시라는 **경영진 주장(assertion)** 축으로 위험 플래그를 설계 → 단순 기능 나열이 아닌 감사 프레임워크 적용.
- **독립 재계산은 감사의 본질이다.** 회사 수치를 그대로 믿지 않고 다시 계산해 차이를 분해하는 것은 실증절차의 핵심이며, 이를 엔진으로 구현했다.
- **근거 추적성(traceability)** 은 감사증적(audit trail)의 디지털 구현이다. "숫자→조항" 링크가 곧 조서의 cross-reference.
- **판단 영역을 자동화로 덮지 않았다.** 선택권·변동리스료·복구의무는 *자동 결론*이 아니라 *감사인에게 던지는 질문*으로 설계 → AI를 보조자로 두는 올바른 태도.
- **디지털 × Assurance의 교집합** 인재상에 부합: 디지털 역량으로 감사 품질을 끌어올리는 사람.

### 예상 질문 & 답변 요지
- *"왜 기존 솔루션과 다른가?"* → 기존은 작성자 도구, 본 프로젝트는 감사인 도구. 근거추적·독립재계산·이상탐지를 더했다.
- *"AI 추출이 틀리면?"* → 그래서 모든 필드에 신뢰도·근거를 붙이고, 저신뢰·누락을 위험 플래그로 강제 검토하게 했다. AI는 결론자가 아니라 1차 추출자.
- *"할인율 기간환산은 왜 두 방식인가?"* → 유효이자율과 명목분할은 리스부채가 달라진다. 가정을 숨기지 않고 노출·전환 가능하게 해 재계산 차이 원인분해를 가능케 했다.

## 8. 향후 확장 로드맵

- ~~**PDF 직접 파싱** 및 스캔 OCR.~~ **(구현됨)** 업로드 PDF의 텍스트 레이어를 서버(`/api/parse-pdf`, pdfjs-dist)에서 추출하고, 텍스트 레이어가 없는 스캔본은 페이지당 문자밀도로 자동 감지해 브라우저 OCR(tesseract.js, 한/영)로 폴백한다. OCR 결과는 인식 오류 가능성을 고려해 **감사 시 원본 대조를 권고**한다. (네이티브 의존성 없이 Vercel 배포 정합 — 서버는 텍스트 추출, OCR은 클라이언트에서 lazy 수행.)
- **포트폴리오 단위 분석** — 다건 일괄 업로드, 회사 전체 리스부채 롤포워드.
- **리스 변경·재측정(modification/reassessment)** 자동 처리 — 할인율 갱신·사용권자산 조정.
- **공시 자동화** — 만기분석·증가표 등 주석 자동 생성, XBRL 프로젝트와 연계.
- **샘플링·테스트 모듈** — 포트폴리오에서 감사 표본 추출.

---

## 9. 한계와 윤리

본 도구는 **감사 보조용**이며 전문가의 판단·검토를 대체하지 않는다. 추출·재계산 결과는 반드시 감사인의 review로 최종화되어야 하며, 규칙기반 임계값(할인율 밴드, 허용오차)은 감사 정책에 맞게 조정되어야 한다.
